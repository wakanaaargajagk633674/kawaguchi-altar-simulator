# 抽出まとめ（source-summary）

> `public/images/shikai/` のスキャン画像から、葬儀司会台本作成アプリの知識ベースを抽出した作業の全体まとめ。

## 読み取り実績
- **読み取った画像枚数**：63枚（すべて JPEG。heic/png/webp は無し）
- **読み取り成功（良好）**：60枚
- **一部不明**：2枚（No.039–040, No.048/050/051 系の薄いページ／本文の一部が [判読不明]）
- **読み取り困難**：1枚（No.049 = `司会マニュアル/SKM_C251i24121615300_0002.jpg`。本文が特に薄く、ショートナレーション例⑥〜⑮付近が判読困難 → 鮮明な再スキャンがあれば補完可能）
- 重複：No.041 と No.042 は同一内容（取材シート記入例）

## 多かった資料種別
1. **フレーズ集（テーマ別ナレーション文例）**：最多。shortphrase 38枚＋ショート50編。趣味・職業・人物像・行事・遺影・季節別の文例＋作成解説。
2. **司会台本（進行台本）**：通夜／告別式（仏式）、無宗教式（お別れの会）。
3. **進行マニュアル**：打合せ・取材・前説の作り方、司会者の心得。
4. **記入様式**：取材シート／司会シート／喪家確認シート（＝アプリ入力フォームの原型）。

## 抽出できた主な式の流れ（詳細は ceremony-flows.md）
- **仏式 告別式**：誘導→前説→ナレ→導師入場→開式の辞(＋初七日)→弔辞→弔電→焼香→導師退場→閉式前ナレ→お別れ準備→閉式→お別れの儀→喪主挨拶→出棺→火葬場同行。
- **仏式 通夜**：誘導→前説→ナレ→導師入場→開式の辞→焼香→導師退場→閉式（「区切り」表現・翌日案内）。
- **無宗教 告別式**：開式ナレ→開式の辞→黙祷→メインナレ→弔辞弔電→思い出映像→お別れの言葉→献奏→献花/焼香→閉式前ナレ→閉式の辞。

## よく使われている表現
- 開式導入：「間もなく開式でございます」「程なくの開式でございます」
- 焼香：「喪主◯◯様よりお進み下さいませ」「順次ご焼香へとお進みくださいませ」
- 導師：「合掌をもってお迎え／お送りくださいませ」「お直り下さいませ」
- 締め：「いつまでも色褪せる事無くご家族の心のアルバムに綴られることと思います」
- 言い換え：ご逝去／永眠／ご生涯を閉じられました（「死亡」を避ける）

## 今後テンプレート化できそうな部分（固定）
前説・誘導・注意事項・導師入退場・開式の辞・黙祷・焼香・献花・献奏枠・弔辞紹介・弔電拝読・導師退場・お別れ準備・閉式・出棺の定型。→ `lib/funeral-script` に文字列テンプレート＋変数で実装（詳細 generation-policy.md / app-template-draft.md）。

## AI生成に任せるべき部分
故人略歴・人柄紹介・遺影写真紹介・趣味エピソード・家族への想い・地域功労・無宗教メインナレ・閉式前ナレ。取材値（入力A群）を素材に、資料由来の文体ルールを制約として生成。

## 固定化すべき部分（再掲・重要）
進行と作法に関わる文言は**必ず固定**。AI に焼香順・宗派作法・初七日有無などを作文させない。生成文に進行案内を混入させないガードを設ける。

## 不明点 / 要追補
- **忌み言葉・避ける表現リスト**：資料に明示なし（業界一般知識で別途用意）。
- **献灯案内**の専用台本：資料が薄い（無宗教式の焼香代替として要追補）。
- **喪主挨拶誘導／精進落とし誘導**の専用台本：断片のみ。要追補。
- No.049 の本文：再スキャン待ち。
- 取材シートの「好きな音楽」など、予定入力項目に無い欄あり（form-fields-draft.md で追加提案済み）。

## 後続のアプリ実装に向けた提案
1. **入力フォーム**を「式の事実（fixed差し込み）」と「取材＝物語素材（AI素材）」に分離（form-fields-draft.md）。
2. **進行は有無フラグで分岐**するテンプレートエンジンを `lib/funeral-script` に。式種別×宗教形式×各フラグでセクション配列を組み立て。
3. **AI生成**は閉じた素材（取材値）＋文体制約プロンプトで、ナレーション系セクションのみ生成。出力は司会者が編集可能に。
4. **HTML→A4印刷**：セクション単位（`break-inside: avoid` for 短文案内 / 長文ナレは改ページ可）。
5. **元画像非依存**：本ディレクトリの Markdown を唯一の知識源とし、`public/images/shikai/` 削除後も動作する設計。
6. 既存 `/iei-photo`（遺影写真作成）と将来連携可能だが、台本機能は**独立した新規ファイル**で追加し既存機能を壊さない（CLAUDE.md 方針）。
7. 個人情報・抽出全文・画像本文を**本番画面に出さない**（解析資料は dev 専用ドキュメント）。
